# 音感トレーニングアプリ 設計・技術スタック方針（初版）

## 1. 目的
本ドキュメントは、MVP 開発に向けた設計方針、技術スタック、インフラ方針、AI 主体の開発方針を現時点の決定事項として整理するものである。

---

## 2. 設計の基本方針

- MVP は **Web アプリ**として実装する
- UI / UX は **スマホブラウザの縦向き利用を最優先**で設計する
- 今後のスマホアプリ展開を見据えつつ、まずは **Web で高速に検証できる構成**を採る
- 開発は **AI 主体**で進める
- 人手レビューは最小限にしつつ、**型・テスト・CI・小さなPR単位**で品質を担保する
- アーキテクチャは、過剰な分散を避けた **モジュラーモノリス** を基本とする

---

## 3. MVP の採用技術スタック

### 3.1 アプリ基盤
- フレームワーク: **Next.js（App Router）**
- 言語: **TypeScript**
- スタイリング: **Tailwind CSS**
- UI コンポーネント方針: 必要に応じて **shadcn/ui 系の考え方を参考**にする

### 3.2 認証
- 認証基盤: **Better Auth**
- ログイン方式: **Google OAuth**
- 認証関連テーブルは Better Auth の推奨構成に従う
- アプリ固有テーブル（training_sessions / question_results / user_settings など）は認証基盤と分離する

### 3.3 データベース / ORM
- データベース: **PostgreSQL**
- ORM / スキーマ管理: **Drizzle ORM**

### 3.4 音源 / 再生
- 再生基盤: **Web Audio API**
- MVP では、**サーバー側で音源ファイルを大量配信する構成は前提としない**
- 出題音は、**クライアント側で生成・再生する方針**を採る
- これにより、配信コストを抑え、再生体験をクライアント側で完結させやすくする
- 音色は MVP では **ピアノ固定** とする

---

## 4. ホスティング / インフラ方針

### 4.1 第一候補
- アプリホスティング: **Vercel**
- データベース: **Neon**

### 4.2 料金方針
- 収益化前の個人開発・非商用フェーズでは、**Vercel Hobby** を利用する
- 収益化または商用利用に入るタイミングで、**Vercel Pro** へ移行する
- DB は当面 **Neon** を利用し、必要に応じて上位プランへ移行する

### 4.3 採用理由
- Next.js との相性が良い
- Preview Deployment により、AI 実装の確認がしやすい
- 運用負荷が低く、個人開発での初期立ち上げに向く
- 自前インフラ運用（VPS / Docker / Firewall / 監視）の負荷を避けられる

---

## 5. リージョン方針

### 5.1 現時点の判断
- Neon には現時点で東京 / 大阪リージョンがないため、**Singapore リージョンを採用する**
- Vercel Functions も、原則として **DB に近いリージョンへ寄せる**

### 5.2 DB レイテンシに対する考え方
本アプリでは、音の再生体験の中心はクライアント側で完結する。
そのため、DB レイテンシは主に以下の処理に影響する。

- ログイン
- 設定保存
- セッション結果保存
- 統計表示

一方で、以下の体験は DB に依存しない設計とする。

- 問題出題中の音再生
- 回答UIの操作
- 回答判定ロジック
- セッション内の一時的な進行状態管理

したがって、MVP では **「回答中に DB を触らない設計」** を徹底することで、Singapore リージョンでも十分成立する構成を目指す。

---

## 6. DB アクセス設計方針

- 問題出題中・回答中は、原則として DB にアクセスしない
- セッション中の回答結果は、一時的にクライアント側またはセッション状態として保持する
- **セッション終了時にまとめて保存**する方針を基本とする
- 設定変更は、UI を即時反映しつつ、保存は非同期で行う
- 統計表示は、必要に応じて集約済みデータや軽量な集計方針を検討する

---

## 7. アプリケーション設計方針

### 7.1 構成方針
アプリ内部は責務ごとに分離し、UI から直接ロジックや永続化に依存しすぎない構造にする。

想定する主要責務は以下の通り。

- auth
- training-config
- audio-engine
- question-generator
- answer-evaluator
- scoring
- session-recorder
- stats
- settings

### 7.2 Server / Client 境界
- 音再生、問題進行、回答UIは **Client 側中心**
- 設定取得、統計取得、保存処理は **Server 側中心**
- 「音が鳴る訓練画面」は client-heavy
- ホーム、設定、結果、統計などは server-first を基本とする

### 7.3 API 方針
- 外部入口が必要なものは **Route Handler** を利用する
- アプリ内部の保存・取得は、MVP では **Server Actions / Server Functions 寄り**でシンプルに構成する
- 不要な REST API の乱立は避ける

## 8. AI 主体の開発方針

### 8.1 基本方針
- 1タスク = 1責務 = 1PR を基本とする
- PR は小さく保つ
- AI に自由実装させる代わりに、**受け入れ条件・型・テスト・CI** で壊れにくくする

### 8.2 品質担保
最低限、以下を自動チェック対象とする。

- format
- typecheck
- lint
- unit test
- DB schema / migration の整合確認

現時点の標準 script は以下。

- `npm run format`
- `npm run lint`
- `npm run lint:repo`
- `npm run check`
- `npm run typecheck`
- `npm run test`
- `npm run build`
- `npm run verify`

`npm run verify` は `typecheck` → `check` → `lint:repo` → `test` → `build` をまとめて実行する。

CI provider は GitHub Actions を採用する。

### 8.3 テスト方針
特に以下は純粋ロジックとして分離し、重点的にテストする。

- 出題ロジック
- 回答判定ロジック
- スコア計算
- 統計集計ロジック

### 8.4 実装順序
MVP では、以下のような縦切りで進める。

1. ゲストでトレーニング開始できる
2. 1問出題して回答できる
3. 結果を表示できる
4. セッション完了まで進められる
5. ログイン時のみ結果保存できる
6. 統計画面を表示できる

## 10. 現時点の意思決定まとめ

現時点では、以下を初期方針として採用する。

- フロントは Next.js App Router + TypeScript
- 認証は Better Auth + Google OAuth
- DB は PostgreSQL + Drizzle ORM
- ホスティングは Vercel
- DB は Neon（Singapore）
- 音は Web Audio API を用いてクライアント側で生成・再生する
- 回答中は DB にアクセスしない
- AI 主体開発を前提に、小さい PR と自動検証で品質を担保する
