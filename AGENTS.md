# AGENTS.md

## 1. このリポジトリの運用方針

このリポジトリは、音感トレーニングアプリ「ミーコピ」の開発用です。

開発の主体は **Codex** です。  
人間は **仕様決定・承認境界・例外対応** のみ担当します。  
ChatGPT は **仕様不足・仕様変更・設計相談・ブロッカー解消** のときだけ使います。

通常の実装・修正・検証・自己修復について、Codex は人間の逐次承認を前提にせず、自律的に進めてください。

---

## 2. Source of Truth

作業前に、以下の順で確認してください。

1. `docs/tasks/active-task.md`
2. `docs/product/current-constraints.md`
3. `docs/product/decision-log.md`
4. `docs/product/requirements.md`
5. `docs/product/basic-design.md`
6. `docs/product/tech-stack.md`
7. `AGENTS.md`
8. 既存コード・型・テスト

### 参照ルール
- 今回の作業範囲・done は `docs/tasks/active-task.md` を最優先する
- 現在有効な固定制約は `docs/product/current-constraints.md` を優先する
- 仕様判断の履歴は `docs/product/decision-log.md` を参照する
- 実装が既存コードと仕様で矛盾する場合は、既存コードではなく仕様ファイルを優先する
- 仕様ファイル同士で矛盾がある場合は、勝手に広げず、より新しく・より具体的な記述を優先する
- それでも判断不能な場合のみ、人間確認事項として明示する

### task ファイルの例外
- 原則として task 正本は `docs/tasks/active-task.md`
- もしそれが存在せず `docs/product/active-task.md` が存在する場合は一時的にそれを参照してよい
- ただし、その場合は人間確認事項に一度だけ明示すること

## 3. Autonomy Policy

### 基本方針
- デフォルトは **自律実行**
- 通常の実装作業で逐次承認を求めない
- 自分でタスクを分解し、小さく安全に進める
- 必要な検証を自分で実行する
- 失敗したら自分で修正を試みる
- 中間確認を求めず、まず完了まで前進する
- 不明点があっても、仕様の範囲で成立する最小実装を優先する
- 「相談した方がよさそう」レベルでは止まらず、まず前に進める

### 止まってよい条件
以下の場合のみ、人間または ChatGPT に相談してよいです。

- Source of Truth に必要な仕様が存在しない
- Source of Truth 同士が矛盾している
- 承認が必要な変更に踏み込む
- 2〜3回の修正ループでも前進できない
- 実装ではなく仕様決定そのものが必要
- データ互換性や意味変更が避けられない

### 止まってはいけないこと
- 単なる実装上の迷い
- 些細な命名相談
- 軽微な設計判断
- 自分で確認できる範囲の検証不足
- 「一応確認したい」という理由だけの停止

---

## 4. Product / UX Fixed Constraints

以下は勝手に変更しないこと。

- MVP は Web アプリとして実装する
- UI / UX はスマホブラウザ縦向きを最優先とする
- 主目的は耳コピにつながる相対音感トレーニングである
- MVP は単音トレーニングを中心とし、基準音ありを前提とする
- 正答率だけでなく、誤差と回答時間を重視する
- 音源は Web Audio API によりクライアント側生成・再生する
- 問題出題中・回答中は DB にアクセスしない
- セッション結果は終了時にまとめて保存する
- ゲスト利用では結果保存しない
- ゲスト結果の後保存・ログイン後バックフィルは行わない
- train 画面はルート内の状態切り替えで進行する
- スコアは内部計算・DB保存ともに小数で扱い、小数第3位まで保持する
- 表示時のみ必要に応じて丸める

詳細は `docs/product/requirements.md` と `docs/product/basic-design.md` を参照すること。

## 5. Architecture Rules

- auth 系テーブルとアプリ固有テーブルは分離する
- Better Auth の `user` を親としてアプリデータを紐づける
- UI と純粋ロジックは分離する
- 純粋ロジックは可能な限り `features/training/model` に寄せる
- training は client-heavy、保存・設定・統計・ホームは server-first を基本とする
- 保存・取得は Server Actions / Server Functions 寄りで構成する
- 不要な REST API は増やさない
- 過剰な分散を避け、モジュラーモノリスを基本とする

---

## 6. Change Policy

### 基本
- 1タスク = 1責務 = 1PR を基本とする
- 無関係な変更を混ぜない
- ついでの大規模リファクタは禁止
- 必要なら段階的に分割する
- 既存コードの流儀は尊重するが、仕様と矛盾する場合は仕様を優先する

### 実装姿勢
- まず最小実装で受け入れ条件を満たす
- 未来の拡張のための過剰設計をしない
- 不要な抽象化を増やさない
- MVP 外の要件を先取りしない
- UX のテンポを落とす処理を入れない

---

## 7. Approval Boundaries

以下は **人間確認が必要** です。  
これらに該当する場合のみ、明示的に確認事項として報告してください。

- 新しい依存ライブラリの追加
- 環境変数の追加・変更
- Better Auth / OAuth / 外部サービス設定の変更
- DB スキーマの意味変更
- migration 適用前提の変更
- データ削除や互換性に影響する変更
- API / URL / 画面導線の大きな変更
- スコア式の意味変更
- 保存項目や分析項目の意味変更
- Source of Truth の不足や矛盾による仕様判断

上記に該当しない限り、原則として人間確認なしで前進してください。

---

## 8. Verification Policy

品質は **人間レビューではなく、受け入れ条件・型・検証・小さい差分** で担保します。

### 期待すること
- 自分で必要な検証を選ぶ
- 失敗したら自分で修正を試みる
- 通るまで前進する
- 通らない場合のみ、何が足りないかを明示する

### 優先検証
- typecheck
- lint
- unit test
- build
- schema / migration 整合確認
- 変更箇所に応じた最小限の追加検証

### 検証コマンド
実際のコマンド名は `package.json` を参照して判断すること。  
`AGENTS.md` に例が書かれていても、それが実態と違う場合は repo 実態を優先してよい。

### 検証失敗時
- まず自分で原因を切り分ける
- 自分で直せるものは直す
- script 未整備など repo 側の不足で進めない場合のみ、その不足を簡潔に報告する

## 9. Reporting Policy

### 最重要原則
**人間は通常、詳細ログを見ない。**  
したがって、報告は **人間確認が必要な事項だけを目立つ形で出す** こと。

### 通常時
人間確認が不要なら、報告は短くてよい。  
探索ログ、閲覧ログ、細かいコマンド履歴、冗長な作業ログは不要。

### 禁止
- `sed`, `rg`, `ls`, `cat` などの探索ログ列挙
- フルコマンド履歴の羅列
- 読んだファイル一覧の羅列
- 実質的に人間が見ない詳細ログの大量出力

### 必要なときだけ報告するもの
- 人間確認が必要な事項
- 受け入れ条件に対して未達がある場合
- 検証基盤が未整備など、repo 側の不足
- 次に自然につながる bundle 候補

---

## 10. Completion Report Format

完了時の報告は、以下の最小形式を使ってください。

### パターンA: 人間確認が必要なことがある場合

```md
### 0. 【要確認】人間に確認してほしいこと
- [確認事項を箇条書き]
- [必要なら理由を短く追記]

### 1. 完了要約
- [何を実装・修正したかを2〜5行で要約]

### 2. 検証と重要操作
- [実行した必須検証の結果]
- [失敗した重要検証があればその理由]
- [migration生成、schema生成、依存追加など重要操作]

### 3. 未解決リスク
- [未解決があれば簡潔に]
- なければ `- なし`

### 4. 次の bundle 候補
- [1〜3件]

### 5. コミットメッセージ案
- [Conventional Commits 形式で1件]
````

### パターンB: 人間確認が不要な場合

```md
### 0. 【要確認】人間に確認してほしいこと
- なし

### 1. 完了要約
- [何を実装・修正したかを2〜5行で要約]

### 2. 検証と重要操作
- [実行した必須検証の結果]
- [必要な重要操作だけ簡潔に記載]

### 3. 次の bundle 候補
- [1〜3件]

### 4. コミットメッセージ案
- [Conventional Commits 形式で1件]
```

### 報告ルール

* 人間確認がないときは、とにかく短くする
* 詳細ファイル一覧は不要
* コマンド履歴の全文は不要
* 「何を変えたか」「確認が必要か」「次に何をやるか」が分かれば十分

---

## 11. Escalation Policy

Codex は以下の順で対応してください。

1. まず仕様正本を読み直す
2. 既存コードと型を確認する
3. 最小実装で進める
4. 必要な検証を回す
5. 自分で修正を試みる
6. それでも止まる場合のみ、人間確認事項として報告する

ChatGPT を前提にしすぎないこと。
まず repo 内の正本で解決を試みること。

---

## 12. Non-Goals

* 人間の逐次レビューを前提にした開発
* 実装のたびに詳細報告を求める運用
* Source of Truth ではなく会話ログに依存する判断
* 仕様未確定のまま大きく広げること
* 不要な確認待ちで開発を止めること

---

## 13. Default Attitude

* まず前に進める
* 迷ったら最小実装
* まず自分で直す
* 必要なときだけ止まる
* 人間の注意を奪いすぎない
* 詳細ログではなく、例外だけを明示する
